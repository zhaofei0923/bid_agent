## 2. Mini-Agent (CLI) 详解

### 2.1 定位

Mini-Agent 是开发任务的**主要执行者**，通过读取标准化的任务文档自动完成开发工作。

### 2.2 核心能力

- **任务解析**: 读取 YAML Front Matter 和 Markdown 正文
- **依赖检查**: 验证前置任务是否完成
- **智能路由**: 根据 `task_type` 分配到专业子代理
- **并行执行**: 无依赖任务可并行处理
- **结果聚合**: 收集验证各子代理产出

### 2.3 子代理类型

| 子代理 | task_type | 专长领域 |
|--------|-----------|---------|
| Coder | `coding` | 代码编写、API实现、组件开发 |
| Tester | `testing` | 单元测试、集成测试、E2E测试 |
| Documenter | `documentation` | 技术文档、API文档 |
| Architect | `design` | 架构设计、数据模型设计 |
| Researcher | `research` | 技术调研、方案对比 |

### 2.4 使用方式

```bash
# 执行单个任务
mini-agent run docs/task-specs/M1-user-system/README.md --task M1-03

# 执行里程碑所有任务
mini-agent run docs/task-specs/M1-user-system/README.md --all

# 仅执行 mini-agent 标记的任务
mini-agent run docs/task-specs/M1-user-system/README.md --executor mini-agent

# 并行执行
mini-agent run docs/task-specs/M1-user-system/README.md --all --parallel

# 查看状态
mini-agent status

# 查看依赖图
mini-agent deps docs/task-specs/M1-user-system/README.md
```

### 2.5 最佳场景

✅ **适合使用 Mini-Agent 的场景**:
- 有标准化任务文档的开发任务
- CRUD API 实现
- React 组件开发
- 单元测试编写
- 批量文件创建/修改
- 数据库迁移脚本生成

❌ **不适合使用 Mini-Agent 的场景**:
- 需要实时交互调试
- 复杂架构决策
- 没有任务文档的临时任务
- 需要多轮讨论的设计问题

## 3. Opus 4.5 (Copilot Chat) 详解

### 3.1 定位

Opus 4.5 是**架构师和审查者**，负责高层设计、任务文档编写和代码质量把关。

### 3.2 核心能力

- **深度推理**: 复杂问题分析和技术方案设计
- **文档生成**: 按模板生成 Mini-Agent 可执行的任务文档
- **代码审查**: 检查安全性、性能、规范符合性
- **疑难诊断**: 分析复杂 bug 和异常行为

### 3.3 使用方式

通过 VS Code GitHub Copilot Chat 访问，选择 **Claude Opus 4.5** 模型。

### 3.4 Prompt 模板

#### 编写任务文档
```
请为 BidAgent 项目编写任务规格文档。

## 功能需求
[描述功能需求]

## 请按照 docs/task-spec-template.md 格式生成，包含：
1. YAML Front Matter (id, executor, priority, task_type, dependencies, outputs)
2. 描述章节
3. 输入/输出文件
4. 验收标准 (AC-N 格式)
5. 详细实现步骤
6. 代码模板

## 技术约束
- 后端: FastAPI + SQLAlchemy 2.x + Pydantic 2.x
- 前端: Next.js 15 + TypeScript + shadcn/ui
- 参考: docs/architecture/data-model.md
- 规范: docs/coding-standards.md
```

#### 架构设计
```
我需要设计 [模块名] 的架构。

## 背景
- 项目技术栈: FastAPI + LangGraph + DeepSeek
- 相关文档: [文档路径]

## 需求
[功能需求列表]

1. 模块目录结构
2. 核心类/接口定义

2. 安全性检查

5. 改进建议
```

#### 疑难诊断

### 3.5 最佳场景

✅ **适合使用 Opus 4.5 的场景**:
- 编写 Mini-Agent 任务文档
- 设计新模块架构
- 数据库表结构设计
- API 接口设计
- 代码质量审查
- 复杂问题诊断
- 技术方案评审

❌ **不适合使用 Opus 4.5 的场景**:
- 直接执行大量文件修改
- 实时调试代码
- 运行终端命令

## 4. Claude Code (独立客户端) 详解

### 4.1 定位

Claude Code 是**补充编码助手**，用于交互式代码修改、实时调试和细节补充。

### 4.2 核心能力

- **交互式编辑**: 实时修改代码并查看效果
- **快速调试**: 分析错误、提供修复并验证
- **细节补充**: 完善 Mini-Agent 遗漏的内容
- **原型验证**: 快速验证技术方案可行性

### 4.3 使用方式

通过独立桌面客户端访问，由 minimax-m2.1 模型驱动。

### 4.4 典型使用场景

#### 补充 Mini-Agent 遗漏
```
Mini-Agent 生成的 auth_service.py 需要补充：

1. 添加邮箱格式验证
2. 密码强度检查需要包含特殊字符要求
3. 添加注册限流逻辑

文件位置: backend/app/services/auth_service.py
请直接修改文件。
```

#### 实时调试
```
运行测试时 test_user_register 失败：

AssertionError: 期望状态码 201，实际 422

请帮我：
1. 分析失败原因
2. 修复代码
3. 重新运行测试验证
```

#### 快速原型
```
我想验证使用 Redis 实现 JWT Token 黑名单的方案：

1. 写一个最小验证代码
2. 运行测试
3. 评估性能
```

### 4.5 最佳场景

✅ **适合使用 Claude Code 的场景**:
- Mini-Agent 执行后需要细节调整
- 实时调试和错误修复
- 快速原型验证
- 交互式代码修改
- 测试覆盖率不足时补充测试

❌ **不适合使用 Claude Code 的场景**:
- 大规模代码生成（使用 Mini-Agent）
- 架构设计（使用 Opus 4.5）
- 需要深度推理的复杂问题（使用 Opus 4.5）

## 6. 最佳实践

### 6.1 高效协作原则

1. **文档先行**: 任何开发任务先确保有任务文档
2. **优先自动化**: 能用 Mini-Agent 自动执行的就不手动
3. **分层审查**: Mini-Agent 执行后，Opus 4.5 审查
4. **及时补充**: 发现问题立即用 Claude Code 修复

### 6.2 任务分解建议

```
大任务
  │
  ├── 架构设计 (Opus 4.5)
  │     └── 输出: 设计文档
  │
  ├── 编写任务文档 (Opus 4.5)
  │     └── 输出: task-spec 文档
  │
  ├── 执行开发 (Mini-Agent)
  │     └── 输出: 代码文件
  │
  ├── 补充修改 (Claude Code)
  │     └── 输出: 完善后的代码
  │
  └── 代码审查 (Opus 4.5)
        └── 输出: 审查报告
```

### 6.3 常见问题处理

| 问题 | 解决方案 |
|------|---------|
| Mini-Agent 执行失败 | 检查任务文档格式，或用 Claude Code 手动修复 |
| 测试覆盖率不足 | 用 Claude Code 补充测试用例 |
| 架构不清晰 | 先用 Opus 4.5 设计架构 |
| 代码不符合规范 | 用 Opus 4.5 审查后指导修改 |
| 实时调试需求 | 使用 Claude Code |

### 6.4 效率提升技巧

1. **批量执行**: 使用 `mini-agent run --all --parallel` 并行执行无依赖任务
2. **复用模板**: Opus 4.5 生成的任务文档可作为后续任务模板
3. **增量开发**: 使用 `--skip-completed` 跳过已完成任务
4. **快速验证**: Claude Code 做原型验证后再写正式任务文档
